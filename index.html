What are Progressive Web Apps?

"Progressive Web Apps use modern web capabilities to deliver an app-like user experience.

They evolve from pages in browser tabs to immersive, top-level apps, maintaining the web's low friction at every moment."

- Addy Osmani
- I think Addy Osmani has a good, succinct explanation - for those who don't know him, he's an engineer at Google and has some famous open source projects like Yeoman and Todo MVC (I personally think we should open a competitor website called TTT OO) - so it's a website that appears like a native app to the user - "modern web capabilities" is an important phrase here but we'll come to that later - they morphe from a normal website to immersive apps

Installing

Characteristics of a PWA

Homescreen access

Full-screen mode

Homescreen access - when a user visits a website a couple of times, they see a prompt to ask if they wish to add the app to the homescreen of their device. This is called a "smart install banner" Full screen mode - when opening the app through tapping on the icon on the homescren, the browser tabs and URL bar disappear, so it looks like you're looking at a native app.

Splash screen - can be implemented to display when the app opens Push notifications - You might know this from Slack when you use it in the browser. It asks you if you want to receive push notifications. With PWAs it's possible to even send push notifications when the app is closed. Android app switcher - When you're switching between apps it looks like an app with its own icon. It doesn't look like it's part of Chrome. Uses HTTPS - so it's very secure. We'll come to it later why it only works with HTTPS Offline - and it finally solves a problem that has existed ever since the internet was invented: you have to be online to use it.

Pros and Cons

Responsiveness üëç

Linkable üëç

No app stores üëç

Browser support üëé

Responsive - as developers we don't need to build different apps for iOS and Android and Windows phone, but only one responsive website is enough. Linkable - as in: it has a URL. Easier to share with others, but what's probably even more important is that search engines can find it and index it. No app stores - from a user perspective it's great because they don't need to download the app but they can just start using it straight away - from a developer perspective we don't need to submit it to the app store and go through approval process but can iterate really quickly. Browser support - there is one big downside at the moment, as with any new shiny front end web technology... and that is browser support. At the moment only on Chrome for Android and Opera for Android. Firefox for Android it's in beta. MS Edge: under development and Webkit: under consideration. Which means for iPhone users: forget it. (For now.) Webkit is a layout engine in browsers. Powers Safari. Chromium based browsers are Chrome and Opera, it's a fork of Webkit.

Service Worker

- a background script run by the browser

- a proxy between your client application and the network

- it can manage a cache of responses

- only works over HTTPS

- can intercept and handle network requests - cannot access the DOM directly but can communicate with your other JavaScript via the postMessage() interface - it is not tied to a specific web page - HTTPS for security reasons. Service workers let you hijack connections and change HTTP responses. Powerful stuff and with great power comes great responsiblity. To make sure it can't be tampered with, you can use it over HTTPS only.

And why would you use it?

You could use it to...

- enhance online performance

- enable offline browsing

- send push notifications

- allow background data synchronisation

Enhanced online experience - by controlling the caching of content, so that it doesn't need to download certain files every time (for example stylesheets or javascript) - could implement a network strategy that optimises the balance between cached content and "fresh content" - fresh content would be your latest blog articles; cached content would be your style sheets and application shell for example. - pre-service worker there was (and still is) something called AppCache. Works in all browsers but Chrome, Firefox are planning to stop support for it. There were issues with it e.g. AppCache can't be partially updated, there were security issues with it and usability problems. Offline experience - you can define what happens when a user tries to use your site or app without a network connection. For example you could have an offline fallback page (the Guardian developer blog has a crossword puzzle) - or you could download certain content that is in the user's click vicinity and if they go offline and click, they still see the pages loading - obviously the user will have to have visited your website before while they were online. Service Worker is magic, but not THAT much magic. Background data synchronisation - service worker can be started up even when the user isn't on the website, so it can update its cache. - but from what I've read this capability is not available yet And what do I hear you screaming?

Lifecycle of a service worker

- if your browser doesn't have a service worker yet for the website, then it needs to connect itself up first. It creates an install event, that we can listen for in our code. - during install step we typically cache some assets like stylesheet or front end JS file - if any files fail to download, the installation step will fail and the service worker won't activate. Good thing: you can trust that your static assets are available when the service worker gets used. - when your browser sees a new service worker, after it installs, it activates it, triggering the activate event. - during the activate phase, clean up of previous versions of the service worker and get rid of old caches - after activation, the service worker will deal with all pages in its remit - then the service worker will be idle and either be terminated to save memory - or it will deal with any fetch events (when a network request is made or a message is received)
